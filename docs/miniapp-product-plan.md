# 单词学习小程序与网页联动产品文档

## 1. 项目背景

当前项目已经具备以下能力：

- 网页端支持单词查询
- 网页端支持我的单词本
- 网页端支持例句练习
- 网页端支持短文学习

下一步希望新增一个微信小程序，主要用于碎片化场景下的查词和收藏单词；网页继续承担更适合长时间学习的练习任务。这样可以形成一个更完整的学习闭环：

- 在小程序里随手查词
- 将单词加入我的单词本
- 在网页里使用这些单词进行例句输入练习
- 后续再扩展复习记录、学习进度和统计能力


## 2. 产品目标

### 2.1 核心目标

- 提供一个轻量、随手可用的查词和收藏入口
- 打通小程序和网页的数据同步
- 让用户在不同设备和场景下都能继续学习
- 逐步沉淀个人单词本和学习记录

### 2.2 产品定位

- 小程序：偏工具型，强调快速查词、快速收藏、快速浏览
- 网页：偏学习型，强调例句练习、短文练习、输入训练

### 2.3 目标用户

- 自己优先使用，作为长期英语学习工具
- 后续如果体验稳定，也可以开放给少量朋友试用


## 3. 使用场景

### 3.1 小程序场景

- 在通勤、排队、碎片时间里快速查一个单词
- 看到陌生词时，立即加入单词本
- 简单浏览自己最近收藏的单词

### 3.2 网页场景

- 在电脑前集中练习单词例句
- 对小程序里添加的单词进行逐个输入训练
- 后续扩展短文、复习计划、学习统计


## 4. 核心产品闭环

一个最重要的闭环如下：

1. 用户在小程序中搜索单词
2. 查看音标、释义、发音、例句
3. 点击加入我的单词本
4. 单词保存到服务端
5. 网页端打开后读取同一份单词本
6. 用户在网页中选择单词进行例句练习
7. 练习结果回写服务端
8. 小程序和网页都能看到该单词的学习状态

这是当前阶段最重要的产品主线，后续开发都应围绕这个闭环展开。


## 5. 功能范围

### 5.1 小程序第一版 MVP

建议优先实现以下功能：

- 微信登录
- 单词搜索
- 单词详情展示
- 发音播放
- 加入我的单词本
- 查看我的单词本
- 从单词本删除单词
- 标记已学 / 未学

### 5.2 网页端需要配合的能力

- 登录后拉取云端单词本
- 按用户维度读取待练习单词
- 从我的单词本进入练习页
- 练习完成后回写学习状态
- 后续增加最近新增、未练习、已掌握等筛选

### 5.3 第二阶段推荐功能

- 小程序展示单词例句
- 小程序显示最近添加记录
- 网页显示学习进度
- 支持收藏时间排序、最近练习排序
- 支持简单复习提醒

### 5.4 第三阶段可扩展功能

- 艾宾浩斯复习策略
- 小程序内做简单拼写练习
- 单词标签分类
- 学习统计与图表
- 多端消息提醒或复习提醒


## 6. 当前项目需要先做的准备

目前网页项目已经有前端能力，但如果要支持小程序和网页联动，必须先补以下基础设施：

### 6.1 数据从本地迁移到服务端

当前部分数据仍然保存在浏览器本地。后续需要将至少以下数据迁移到服务端：

- 我的单词本
- 用户学习状态
- 练习记录
- 自定义内容（后续再决定是否上云）

### 6.2 增加用户体系

因为小程序和网页要共用数据，所以必须知道“谁是同一个用户”。建议至少有：

- 用户唯一 ID
- 登录来源
- 账号创建时间

### 6.3 统一 API 层

前端页面不应直接依赖本地存储逻辑，而应逐步改为调用 API。这样小程序和网页才能共用同一套后端能力。

### 6.4 统一数据模型

需要提前规范以下对象结构：

- 用户
- 单词
- 用户单词本
- 例句
- 学习记录


## 7. 技术方案建议

### 7.1 总体架构

建议采用以下结构：

- 小程序前端：微信小程序
- 网页前端：当前 React 项目
- 后端服务：部署在腾讯云轻量服务器
- 数据库：MySQL 或 PostgreSQL
- 反向代理：Nginx
- HTTPS：使用 Nginx + SSL 证书

### 7.2 为什么适合用腾讯云轻量服务器

你目前已经有腾讯云轻量服务器，这对当前阶段很合适：

- 可直接部署 Node.js 后端服务
- 可部署数据库或先连接云数据库
- 可统一承载网页 API 和小程序 API
- 成本可控，适合个人项目迭代

### 7.3 推荐部署方式

建议将系统拆成两部分：

- 前端网页：继续走当前静态站点构建方式
- 后端服务：单独一个 Node.js API 服务

服务器建议部署内容：

- `Nginx`
- `Node.js`
- `PM2`
- `MySQL` 或 `PostgreSQL`

推荐目录思路：

```text
/srv/words-app/web
/srv/words-app/api
/srv/words-app/logs
```

### 7.4 后端建议

建议优先做一个简单统一后端，供网页和小程序共用。

推荐能力：

- 用户登录鉴权
- 查词代理接口
- 单词本增删改查
- 例句读取
- 学习记录回写

如果后端技术保持一致，后续维护成本会更低。

## 9. API 规划建议

建议第一阶段先做以下接口。

### 9.1 用户相关

- `POST /api/auth/wechat-login`
- `GET /api/user/profile`

### 9.2 单词查询

- `GET /api/words/search?keyword=...`
- `GET /api/words/:id`

### 9.3 我的单词本

- `GET /api/wordbook`
- `POST /api/wordbook`
- `DELETE /api/wordbook/:id`
- `PATCH /api/wordbook/:id`

### 9.4 例句与练习

- `GET /api/words/:id/sentences`
- `POST /api/study-records`
- `GET /api/study-records/summary`

### 9.5 可选接口

- `POST /api/words/batch-import`
- `GET /api/wordbook/recent`
- `GET /api/wordbook/review-list`

## 11. 开发优先级

### 阶段 1：打基础

- 设计数据库
- 搭建后端服务
- 部署到腾讯云服务器
- 完成登录和鉴权

### 阶段 3：开发小程序

- 搭建小程序项目
- 完成查词页

- 完成我的单词本页

### 阶段 4：完善学习闭环

- 回写练习记录
- 显示学习状态
- 增加最近新增和待复习列表


## 12. MVP 验收标准

当以下能力都能跑通时，可以认为第一版闭环成立：

- 用户能在小程序中登录
- 用户能搜索单词
- 用户能将单词加入单词本
- 单词本数据保存到服务端
- 网页能读取同一个用户的单词本
- 网页能基于这些单词进入例句练习
- 练习结果能回写服务端


## 13. 风险与注意事项

### 13.1 当前主要风险

- 如果继续依赖本地存储，将无法实现跨端同步
- 如果没有统一登录体系，无法识别同一用户
- 如果查词接口完全依赖第三方服务，稳定性需要重点关注
- 如果接口设计不统一，后续小程序和网页维护成本会升高

### 13.2 建议策略

- 先把“单词本”上云，再考虑更多学习功能
- 先做最小闭环，不要一开始做过多页面
- 接口设计尽量保持稳定，避免前后端频繁返工
- 第三方词典和翻译能力做好缓存与兜底


## 14. 近期开发建议

基于当前项目，建议近期按以下顺序推进：

1. 把我的单词本从本地存储迁移到服务端
2. 设计数据库和基础 API
3. 在腾讯云轻量服务器上部署后端服务
4. 网页端改为读取云端单词本
5. 开始开发小程序查词和收藏功能
6. 最后补学习记录与同步状态


## 15. 一句话总结

当前最正确的方向不是先堆很多小程序页面，而是先完成“云端单词本 + 用户体系 + 统一 API”这三件基础工作。只要这三项打通，小程序负责查词收藏、网页负责练习学习的产品路线就能稳定成立。
